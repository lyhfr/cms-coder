# 工具治理与权限控制设计

## 1. 文档目标

定义 cmscoder 对本地工具调用的风险识别、前置检查、交互确认和策略执行机制。

## 2. 关联总纲

- 来源章节：6.1、8.1.6、10.4、12、13.1、14.1、18
- 相关文档：[plugin-architecture.md](./plugin-architecture.md)、[../shared/quota-audit-observability.md](../shared/quota-audit-observability.md)

## 3. 设计范围

- 工具调用分类与风险分级
- allow / ask / deny 策略模型
- Bash、文件写入、代码修改、测试执行等高风险操作治理
- 策略命中后的提示、阻断、审计回传

## 4. 关键场景或流程

- 运行危险命令前的确认
- 写文件或修改代码前的策略判断
- 测试执行、批量改动、删除操作的风险拦截
- 企业后续接入审批流时的扩展接口

## 5. 设计要点

- 工具治理首先发生在插件端前置层
- 高风险操作需支持提醒或阻断
- 策略模型需要兼容用户、团队、项目、环境等维度
- 治理结果应可被服务端审计或统计

## 6. 接口、数据或配置

- 策略配置：风险等级、命令规则、路径规则、环境规则
- 运行数据：命中的策略、用户确认结果、执行结果、异常原因
- 待补充：是否由服务端下发策略，以及策略的缓存与刷新机制

## 7. 非功能要求

- 策略判断延迟低，不明显影响使用体验
- 规则可扩展，支持新增工具类型
- 治理日志可追踪、可审计

## 8. 风险与待确认

- 工具调用拦截点在不同 Agent 中是否一致
- 本地策略与服务端策略冲突时的优先级待明确
- 是否需要支持离线兜底策略待确认

## 9. 验收标准

- 高风险工具调用可在执行前被识别和处理
- allow / ask / deny 策略可按用户、团队、项目生效
- 治理结果能够被记录并支持后续审计
