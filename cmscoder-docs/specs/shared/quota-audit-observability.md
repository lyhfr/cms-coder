# 配额、审计与可观测设计

## 1. 文档目标

定义 cmscoder 在额度控制、限流、审计留痕和端到端观测上的设计边界。

## 2. 关联总纲

- 来源章节：6.1、8.2.7、8.2.8、13.3、14.2、16、18
- 相关文档：[../web-server/server-architecture.md](../web-server/server-architecture.md)、[../plugin/tool-governance-permission-control.md](../plugin/tool-governance-permission-control.md)

## 3. 设计范围

- 用户、团队、项目三级额度控制
- RPM / TPM / 并发限制
- 请求日志、错误日志、模型使用量审计
- request id / trace id 追踪
- 插件端可见的额度摘要和异常状态

## 4. 关键场景或流程

- 请求进入服务端时命中额度与限流规则
- 模型调用完成后记录使用量与审计日志
- 上游异常或限流异常被追踪和展示
- 插件端读取额度摘要和故障状态

## 5. 设计要点

- 配额和审计主逻辑位于服务端
- 插件端主要承担状态展示和诊断入口
- 审计要覆盖请求、错误、模型消耗和关键治理动作
- 观测体系需要支持后续对接企业审计系统

## 6. 接口、数据或配置

- 配额配置：用户额度、团队额度、项目预算、默认限流规则
- 审计数据：用户、团队、项目、模型、时间、request id、trace id、结果状态
- 插件端展示：可用模型、默认模型、额度摘要、异常信息

## 7. 非功能要求

- 观测链路完整，支持问题回溯
- 审计数据具备合规性与可保留性
- 限流与配额判定稳定、可解释

## 8. 风险与待确认

- 审计日志的保留时长与脱敏策略待确认
- 配额是否需要支持部门级或成本中心维度待确认
- 插件端可展示的额度粒度和实时性待明确

## 9. 验收标准

- 服务端可按用户、团队、项目维度控制额度和查看消耗
- 每次模型调用都可通过 request id / trace id 追踪
- 插件端能展示额度摘要和关键异常状态
